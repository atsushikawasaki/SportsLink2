# SportsLink 完全要件定義書

## 1. プロジェクト概要

### 1.1 システム名

SportsLink（スポーツ大会運営支援システム）

### 1.2 目的

ソフトテニス競技における大会運営を支援するシステム。リアルタイムスコアリング、オフライン対応、マルチロール対応、観戦者向け公開といった大会実用性を重視したWebアプリケーション。

### 1.3 主要機能

- リアルタイムスコアリング（Supabase Realtime経由）
- オフライン対応（Dexie/IndexedDB + 自動同期）
- 競合解決（409 Conflict / 楽観的ロック）
- RBAC権限管理（user_permissionsテーブル + RLS）
- 監査ログ（5年間保持）
- 大会管理・ドロー生成（シングルエリミネーション）
- エントリー管理（CSVインポート/エクスポート）
- 当日受付（チェックイン）と認証キー（4桁）発行
- PDF/CSV 結果エクスポート
- Sentryによるエラーモニタリング

### 3.2 データ変更フロー

- **Server Actions**（`src/app/actions/`）: UIからの直接呼び出し。`'use server'`ディレクティブ使用。Zodバリデーション後にSupabaseへ書き込み、`revalidatePath`でキャッシュ無効化。
- **API Routes**（`src/app/api/`）: REST形式。スコアリングなどオフライン同期が必要な操作に使用。`createClient()`でサーバーサイドSupabaseクライアントを生成。
- **TanStack Query**（`src/lib/hooks/`）: データフェッチング・キャッシュ管理。ミューテーション時に楽観的更新を行う場合もある。

### 3.3 セキュリティ対策

#### HTTPレスポンスヘッダー（`next.config.mjs`）

```
X-Frame-Options: DENY
X-Content-Type-Options: nosniff
Referrer-Policy: strict-origin-when-cross-origin
Permissions-Policy: camera=(), microphone=(), geolocation=()
X-XSS-Protection: 1; mode=block
```

#### レート制限（`src/lib/rateLimit.ts`）

- インメモリ Map ベース（IPアドレスキー）
- ウィンドウ: 60秒 / 最大試行: 10回
- 適用エンドポイント: `login`, `signup`, `forgot-password`, `update-password`
- X-Forwarded-For ヘッダーでIPを取得

#### その他

- APIレスポンスから `password_hash` を除外（`sanitizeUser()` / `omitPasswordHash()` ヘルパー）
- `markdown-loader.ts` の `version` パラメータにパストラバーサル対策（正規表現 `^\d+\.\d+(\.\d+)?$`）
- CSRF トークン（`src/lib/csrf.ts`）: `csrf_token` Cookie から取得し `X-CSRF-Token` ヘッダーで送信
- べき等性保証（`src/lib/idempotency.ts`）: `Idempotency-Key` ヘッダーを用いた24時間TTLキャッシュ

### 4.3 RLS（Row Level Security）

- Supabase Auth の `auth.uid()` を使って各テーブルへのアクセスを制御
- 管理操作には `createAdminClient()`（service_role key）を使ってRLSをバイパス
- `user_permissions` テーブルでロール確認後に操作を許可

### 4.4 監査ログトリガー

- points, matches, match_scores, tournaments, teams, users, user_permissions の変更を自動記録
- 5年間保持（`cleanup_old_audit_logs` 関数）

## 5. 権限管理

### 5.1 権限確認ユーティリティ（`src/lib/permissions.ts`）

```typescript
type RoleType = 'admin' | 'tournament_admin' | 'team_admin' | 'umpire';

// 汎用権限チェック
checkPermission(userId, roleType, scope?: { tournament_id?, team_id?, match_id? }): Promise<boolean>

// ショートハンド
isAdmin(userId): Promise<boolean>
isTournamentAdmin(userId, tournamentId): Promise<boolean>
isTeamAdmin(userId, teamId): Promise<boolean>
isUmpire(userId, matchId): Promise<boolean>
```

- `role_type='admin'` は全権限を持つ（スコープ問わず）
- `checkPermission` は内部で `createAdminClient()`（RLSバイパス）を使用

### 5.2 Server Actions での権限チェック

Server Actions（`src/app/actions/`）では、Supabaseの `supabase.auth.getUser()` で認証確認後、`checkPermission` / `isAdmin` / `isTournamentAdmin` で認可チェックを行う。

### 5.3 APIルートでの権限チェック

API Routes（`src/app/api/`）でも同様に `createClient().auth.getUser()` で認証し、必要に応じて `checkPermission` を呼び出す。

## 6. APIエンドポイント

### 6.1 API共通仕様

- **認証**: Supabase Cookie-based セッション（`createClient()` がCookieを自動参照）
- **エラーレスポンス形式**: `{ error: string, code: string, details?: unknown }`
- **ページネーション**: `limit`（デフォルト10）、`offset`（デフォルト0）

#### エラーコード一覧

| コード | HTTPステータス | 説明 |
|---|---|---|
| `E-AUTH-001` | 401 | 認証失敗（メール/パスワード不一致、未認証） |
| `E-AUTH-002` | 500 | セッション作成失敗 |
| `E-AUTH-003` | 500 | 認証ユーザー作成失敗 |
| `E-RATE-001` | 429 | レート制限超過 |
| `E-VER-003` | 400 | バリデーションエラー |
| `E-CONFL-001` | 409 | 楽観的ロック競合（version不一致） |
| `E-SYNC-PENDING` | 409 | 未同期オフラインキューがある状態でのUndo/確定操作 |
| `E-RLS-002` | 403 | RLSによるアクセス拒否 |
| `E-NOT-FOUND` | 404 | リソース不存在 |
| `E-DB-001` | 500 | データベースエラー |
| `E-SERVER-001` | 500 | サーバー内部エラー |

### 6.2 認証（`/api/auth`）

| メソッド | パス | 説明 |
|---|---|---|
| POST | `/api/auth/login` | ログイン（レート制限: `login`） |
| POST | `/api/auth/signup` | サインアップ（レート制限: `signup`） |
| POST | `/api/auth/forgot-password` | パスワードリセットメール送信 |
| POST | `/api/auth/reset-password` | パスワードリセット（トークン検証後） |
| PUT | `/api/auth/password` | パスワード変更（レート制限: `update-password`） |
| PUT | `/api/auth/profile` | プロフィール更新 |
| GET | `/api/auth/me` | 現在のユーザー情報取得 |
| GET | `/api/auth/users` | ユーザー一覧取得（管理者用） |
| DELETE | `/api/auth/account` | アカウント削除 |
| GET | `/api/auth/consent/check` | 規約同意状況チェック |

### 9.2 認証フロー

#### AuthProvider（`src/components/AuthProvider.tsx`）

- アプリ全体をラップするClient Component
- 初期化時に `supabase.auth.getSession()` でセッション確認し、`useAuthStore` にセット
- `onAuthStateChange` で認証状態変化を監視
- `hasHydrated` で非同期ハイドレーション待ちを管理

#### ログインフロー

1. POST `/api/auth/login`（メール・パスワード）
2. `supabase.auth.signInWithPassword()` 呼び出し
3. 失敗時: `public.users` の `password_hash` を bcryptjs で検証（フォールバック）
4. 成功時: セッションCookieを設定し、ユーザー情報を返却
5. レスポンスから `password_hash` を除外（`sanitizeUser()` で処理）
6. `GET /api/auth/consent/check` で規約同意状況を確認
7. 未同意 → `/consent`、同意済み → `/dashboard`

#### サインアップフロー

1. POST `/api/auth/signup`（displayName, email, password, agreeTerms）
2. レート制限チェック（`signup`）
3. パスワード最低8文字チェック
4. `supabase.auth.signUp()` 呼び出し
5. DBトリガー `handle_new_user` が `public.users` レコードを自動作成
6. 同意ログを `user_consents` に記録
7. 成功後 `/login?registered=true` にリダイレクト

## 10. スコアリングロジック

### 10.1 スコア計算（`src/lib/scoring/aggregateMatchScore.ts`）

- `points` テーブルから `is_undone=false` のポイントを集計し `game_count_a` / `game_count_b` を計算
- フロントエンドは `useMatchStore` の `gameCountA`, `gameCountB`, `currentScoreA`, `currentScoreB` でリアルタイム表示

### 10.2 スコアリングルール

#### 通常ゲーム

- スコア進行: 0 → 1 → 2 → 3 → Game
- ゲーム終了条件: 4ポイント先取 & 2ポイント差
- デュース: 3-3になった場合
- アドバンテージ: デュース後1ポイント差（例: AD-3）

#### ファイナルゲーム（タイブレーク）

- 開始条件: `gamesToWin - 1` 同点時
- 終了条件: 7ポイント先取 & 2ポイント差
- `gamesToWin`: `Tournament_Phases.config.gamesToWin` から取得（デフォルト: 4）

### 10.3 楽観的ロック

- `matches.version` を使用
- ポイント追加・ステータス更新時に `version` が一致しない場合は `409 E-CONFL-001`
- 更新成功時は `version` を `currentVersion + 1` に更新

### 10.4 Undo機能

- 最新ポイントを `is_undone=true` に更新（論理削除）
- スコアを再計算し `match_scores` を更新
- `version` をインクリメント
- 未同期キューが存在する場合は `E-SYNC-PENDING` エラー

## 11. ドロー生成ロジック

### 11.1 生成手順

1. `tournament_entries` からエントリー取得
2. シード配置（優先順位: `seed_rank` 指定 > `performance_score` > 所属分散 > 番手分散）
3. ブラケットサイズを2の累乗に拡張し、BYEを設定
4. `matches`, `match_slots` レコードを作成

### 11.2 BYE判定

- エントリー数が2の累乗でない場合に発生
- `match_slots.source_type = 'bye'` で表現
- BYE の試合は自動的に相手側を次ラウンドに進める

### 11.3 ドロー構造

- Phase → Rounds → Matches → Slots の階層構造
- `match_slots.source_type`: `entry`（確定エントリー）/ `winner`（前試合の勝者）/ `loser`（前試合の敗者）/ `bye`（不戦勝）

- `NotificationType`: `'auth_key'` / `'umpire_assignment'` / `'match_waiting'` / `'system'`
- `getAuthKey(tournamentId?)` で特定大会の認証キーを取得可能

### 15.2 DB通知（`notifications` テーブル）

- `NotificationType`: `'auth_key'` / `'match_start'` / `'umpire_assignment'` / `'tournament_update'`
- 未読管理（`is_read`）

### 15.3 NotificationCenter

- ヘッダー右側のベルアイコン
- `useNotificationStore` から通知を取得して表示
- 未読バッジ表示
- 通知クリックで関連ページに遷移

## 16. 設定画面（`/settings`）

### 16.1 タブ構成

- **プロフィール**: 表示名編集、メール表示（`PUT /api/auth/profile`）
- **セキュリティ**: パスワード変更（`PUT /api/auth/password`）、アカウント削除（`DELETE /api/auth/account`）
- **通知設定**: UIのみ実装（永続化未実装）
- **アプリ設定**: UIのみ実装（永続化未実装）

## 17. エラーハンドリング

### 17.1 フロントエンド

- `ErrorBoundary`: Reactコンポーネントエラーをキャッチし、ユーザーフレンドリーなメッセージを表示
- Toast通知: 操作成功・失敗のフィードバック
- 競合解決モーダル: `E-CONFL-001` 発生時に最新データを再取得して表示
- 接続状態表示: ConnectionStatus コンポーネント

### 17.2 Sentry（`@sentry/nextjs`）

- 環境変数: `NEXT_PUBLIC_SENTRY_DSN`, `SENTRY_ORG`, `SENTRY_PROJECT`
- `next.config.mjs` の `withSentryConfig` でラップ
- `widenClientFileUpload`, `hideSourceMaps`, `disableLogger` オプション設定済み
